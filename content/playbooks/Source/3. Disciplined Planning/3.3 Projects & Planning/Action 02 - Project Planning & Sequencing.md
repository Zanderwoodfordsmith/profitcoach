# Project Planning & Sequencing

## What to Do

Take each prioritised project and break it into a clear plan with milestones, dependencies, and a realistic timeline. Then sequence your projects so they support each other rather than competing for the same people and resources at the same time.

This is not about building elaborate Gantt charts. It is about answering three questions for every project: What are the key milestones? What depends on what? And when does each piece realistically happen given everything else on the plate?

## Why It Matters

Most projects fail not because the idea was bad but because the plan was vague. "Launch the new service" is not a plan. It is a wish. Harnish's Scaling Up methodology insists on breaking rocks into specific, time-bound milestones because that is where accountability becomes real. And the sequencing piece matters because dependencies are invisible until they cause problems. Project B cannot start until Project A delivers a key output, but nobody mapped that, so both stall.

## How to Use This

Work through this for each of your top 3-5 projects from the Strategic Project Prioritisation tool. Start with the highest-priority project and work down.

---

## Step 1: Define Project Scope

- [ ] **Write a one-sentence project objective** - "By [date], we will have [specific deliverable] so that [business outcome]." If you cannot write this sentence, the project is not clear enough to plan.
- [ ] **List what is in scope and what is not** - Scope creep kills projects. Be explicit about boundaries now so you do not argue about them later.
- [ ] ⚡ Quick Win: **Define the "done" criteria** - What does completion actually look like? What can you see, touch, or measure that proves this project is finished?

## Step 2: Break into Milestones

- [ ] **Identify 3-6 key milestones** - Major checkpoints between start and finish. Each milestone should be a concrete deliverable, not a vague phase like "planning" or "development."
- [ ] **Estimate duration for each milestone** - In weeks, not days. Be realistic. Whatever you think, add 30% for the unexpected.
- [ ] **Identify dependencies between milestones** - Which milestones must complete before others can start? Draw these links explicitly. This is where most hidden delays live.
- [ ] **Flag external dependencies** - Anything that depends on a supplier, client, or third party. These are your highest-risk items because you do not control the timeline.

## Step 3: Sequence Across Projects

- [ ] **Map all projects on a single timeline** - Use a wall, whiteboard, or simple spreadsheet. Seeing everything together reveals clashes that are invisible when you plan projects in isolation.
- [ ] ⚡ Quick Win: **Identify resource collisions** - Where do two projects need the same person or team at the same time? Shift one. Do not pretend people can do both.
- [ ] **Stagger project starts** - Launching three projects in the same week guarantees none of them get proper attention. Space starts by 2-4 weeks minimum.
- [ ] **Build in buffer between dependent projects** - If Project B depends on Project A finishing, do not schedule them back-to-back. Leave at least one week of breathing room.
- [ ] **Set monthly review checkpoints** - A standing 30-minute slot to review all project timelines together. This is where you catch drift early rather than discovering it at the deadline.

---

## Expected Outcome

- Each priority project has a clear objective, scope, and done criteria
- Milestones and dependencies mapped for every active project
- A single view showing all projects across the same timeline
- Resource collisions identified and resolved before they cause problems
- A realistic sequence that gives each project the attention it needs

## Coach Notes

Clients often want to skip straight to execution. Push them to spend the time on planning - the quality of the plan directly predicts whether the project finishes on time or drags on for months.

The most useful question during this exercise is "What could stop this milestone from completing on time?" It surfaces risks that the client is aware of but has not articulated. Once they are on paper, you can plan around them.

If a client has never planned projects this way before, start with just one project as a worked example. Once they see the value, they will want to do the rest.
